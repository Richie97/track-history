Getting started

Track History is a single Cloudflare Worker (Hono API + static frontend) backed by a D1 SQLite database. This page covers running it locally and deploying your own instance — both fit comfortably in Cloudflare's free tier.

Local development

git clone https://github.com/richie97/track-history
cd track-history
npm install
npm run seed:generate       # writes seed/seed.sql from your seed data (see below)
npm run db:migrate:local    # creates the local SQLite schema
npm run db:seed:local       # loads the seed data
npm run dev                 # http://localhost:8787

Create a .dev.vars file (gitignored) in the project root:

DEV_MODE=1
DEV_USER_EMAIL=you@example.com   # match your seed data's USER_EMAIL
DEV_USER_NAME=Your Name
GOOGLE_CLIENT_ID=dev
GOOGLE_CLIENT_SECRET=dev

DEV_MODE=1 replaces Google login with a fixed local dev user, so you can develop without OAuth credentials — open /auth/login once to get a session cookie.

Never set DEV_MODE in production. It bypasses authentication entirely.

Seeding your own history

seed/generate.mjs reads seed/data.personal.mjs if it exists (gitignored — your real name, email and lap history stay out of the repo), otherwise seed/data.example.mjs. Copy the example file to data.personal.mjs, fill in your events, and re-run npm run seed:generate.

Deploying to Cloudflare

1. Log in & create the database

npx wrangler login
npx wrangler d1 create track-history

Copy the database_id it prints into wrangler.jsonc (replacing the zeros).

2. Apply the schema & seed your history

npm run db:migrate:remote
npm run db:seed:remote      # imports your history (run once)

3. Create a Google OAuth client

  • Go to console.cloud.google.com/apis/credentials (any project).
  • Create credentials → OAuth client ID → Web application.
  • Authorized redirect URI: https://track-history.<your-subdomain>.workers.dev/auth/callback. Run npx wrangler deploy once first if you don't know your workers.dev subdomain, and add your custom domain's /auth/callback too if you attach one.
  • If prompted to configure the consent screen: External, add yourself as a test user — or publish it, since only people you expect can do anything anyway.

4. Set secrets & deploy

npx wrangler secret put GOOGLE_CLIENT_ID
npx wrangler secret put GOOGLE_CLIENT_SECRET
npm run deploy

Sign in with the Google account matching your seed data's USER_EMAIL and it claims the imported history automatically. Other Google accounts get a fresh, empty logbook — the instance is multi-user, and every row is scoped to its owner.

Running the tests

npm test              # vitest: unit tests + API tests against a real D1 (miniflare)
npm run typecheck     # tsc over src/ and test/

API tests run the whole Worker under @cloudflare/vitest-pool-workers against a real D1 database with migrations applied per test, so ownership isolation between users is tested end to end.
