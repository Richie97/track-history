Data model & stats

The database mirrors the shape of a season: tracks → events → sessions → laps. Everything below is derived from that, with a couple of deliberate rules worth knowing.

The hierarchy

  • Track — identified by name and configuration (e.g. full course vs. short course), matched case-insensitively. A layout change is a different track for record-keeping, so best times and goals never mix across configurations.
  • Event — a day (or weekend) at a track, with notes and an optional manual best time.
  • Session — one run group / stint within an event.
  • Lap — an integer number of milliseconds. Lap times are milliseconds everywhere in the system; the frontend formats and parses m:ss.fff.

Best time

An event's best time is MIN(best logged lap, manual best). The manual field exists because spreadsheet-era events often only recorded a single best time — you can import those as-is and attach full lap data later, either via the seed data or pasted into the UI. Imported per-session bests appear as one-lap sessions.

Consistency

Consistency is the coefficient of variation (standard deviation ÷ mean) of all laps in an event, shown once an event has 3+ laps. Lower is more consistent. Per-session analysis also derives best-N averages, a pace slope over the stint, and warmup-lap detection.

Share pages

You can publish your logbook read-only under a slug you choose (/share/<slug>). The public endpoint returns stats and times but strips notes, your email, and per-lap data — it's the only route that skips authentication.

Architecture in one paragraph

One Cloudflare Worker serves both the JSON API (Hono) and the static frontend. The frontend is dependency-free vanilla JS — native ES modules, a hash router, template-string HTML and hand-rolled SVG charts, with no build step. The database is D1 (SQLite) with ordered migrations. A service worker makes the app installable and usable offline: static files are served stale-while-revalidate, navigations and API GETs are network-first with cache fallback.

Privacy posture

  • Self-hosted: your instance, your Cloudflare account, your data.
  • Multi-user with ownership scoping on every row; another Google account signing in gets its own empty logbook.
  • Telemetry and video parsing is entirely client-side — recordings are never uploaded.
  • Personal seed data (seed/data.personal.mjs) is gitignored so your real history never lands in the repo.
